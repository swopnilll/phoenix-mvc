# Phoenix Request Lifecycle: From Browser to Response

## The Journey of a Request

When a browser hits your Phoenix app, the request flows through a pipeline:

```
Browser Request
     ↓
┌─────────────────────────────────────────────────────┐
│  ENDPOINT (lib/hello_web/endpoint.ex)               │
│  ├── Plug.Static      → Serve CSS, JS, images      │
│  ├── Plug.RequestId   → Add unique request ID      │
│  ├── Plug.Parsers     → Parse JSON, form data      │
│  ├── Plug.Session     → Handle cookies/sessions    │
│  └── HelloWeb.Router  → Route to handler           │
└─────────────────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────────────────┐
│  ROUTER (lib/hello_web/router.ex)                   │
│  ├── pipe_through :browser  → Add browser plugs    │
│  └── Match route:                                   │
│      • get "/home" → Controller (one-time response)│
│      • live "/products" → LiveView (persistent)    │
└─────────────────────────────────────────────────────┘
     ↓
┌─────────────────┐     OR     ┌─────────────────────┐
│   CONTROLLER    │            │      LIVEVIEW       │
│   (one-time)    │            │   (persistent)      │
│                 │            │                     │
│  Request → HTML │            │  Request → HTML     │
│  Connection     │            │  + WebSocket opens  │
│  CLOSES         │            │  Connection STAYS   │
└─────────────────┘            └─────────────────────┘
```

## What is the Endpoint?

The **Endpoint** (`lib/hello_web/endpoint.ex`) is the entry point for ALL requests. It's a series of "plugs" (middleware) that process the request before it reaches your routes.

```elixir
defmodule HelloWeb.Endpoint do
  use Phoenix.Endpoint, otp_app: :hello

  # LiveView WebSocket connection (for real-time features)
  socket "/live", Phoenix.LiveView.Socket,
    websocket: [connect_info: [session: @session_options]]

  # Serve static files from priv/static
  plug Plug.Static,
    at: "/",
    from: :hello

  # Parse JSON, form data, file uploads
  plug Plug.Parsers,
    parsers: [:urlencoded, :multipart, :json]

  # Handle sessions via cookies
  plug Plug.Session, @session_options

  # Finally, send to router
  plug HelloWeb.Router   # ← This is where YOUR code takes over
end
```

**Key insight:** Every plug transforms the `conn` (connection) struct and passes it to the next plug. Think of it as an assembly line.

## Option 1: Traditional Controller (Request-Response)

Use controllers when you need **one-time responses**: static pages, file downloads, API endpoints.

### Step 1: Define the Route

In `lib/hello_web/router.ex`:

```elixir
scope "/", HelloWeb do
  pipe_through :browser   # Use browser-specific plugs

  get "/home", PageController, :home
  #    │         │              │
  #    │         │              └── Function to call
  #    │         └── Controller module
  #    └── URL path
end
```

The `pipe_through :browser` adds plugs for:
- CSRF protection (`:protect_from_forgery`)
- HTML format (`:put_secure_browser_headers`)
- Session handling

### Step 2: Create the Controller

In `lib/hello_web/controllers/page_controller.ex`:

```elixir
defmodule HelloWeb.PageController do
  use HelloWeb, :controller

  def home(conn, _params) do
    render(conn, :home)   # Renders home.html.heex template
  end
end
```

The `conn` is the connection struct containing:
- `conn.params` - Query params and route params
- `conn.assigns` - Data you assign for templates
- `conn.cookies` - Cookie data

### Step 3: Create the Template

In `lib/hello_web/controllers/page_html/home.html.heex`:

```heex
<h1>Welcome Home!</h1>
```

## Real Example: File Download Controller

A practical use case for controllers - downloading files:

### Route

```elixir
scope "/", HelloWeb do
  pipe_through :browser

  get "/download/:filename", DownloadController, :download
end
```

### Controller

```elixir
defmodule HelloWeb.DownloadController do
  use HelloWeb, :controller

  def download(conn, %{"filename" => filename}) do
    file_path = Path.join(["priv", "downloads", filename])

    if File.exists?(file_path) do
      conn
      |> put_resp_content_type("application/octet-stream")
      |> put_resp_header("content-disposition", ~s(attachment; filename="#{filename}"))
      |> send_file(200, file_path)
    else
      conn
      |> put_flash(:error, "File not found")
      |> redirect(to: ~p"/")
    end
  end
end
```

**Why use a Controller here?**
- File download is a one-time action
- No need for real-time updates
- Response is binary data, not interactive HTML

## Option 2: LiveView (Real-Time Connection)

Use LiveView when you need **interactive, real-time** UIs.

### Step 1: Define the Route

```elixir
scope "/", HelloWeb do
  pipe_through :browser

  live "/products", ProductsLive
  #     │            │
  #     │            └── LiveView module (not Controller!)
  #     └── URL path
end
```

### Step 2: Create the LiveView

In `lib/hello_web/live/products_live.ex`:

```elixir
defmodule HelloWeb.ProductsLive do
  use HelloWeb, :live_view

  def mount(_params, _session, socket) do
    # Called when page loads - like componentDidMount in React
    products = Hello.Store.list_products()
    {:ok, assign(socket, products: products)}
  end

  def render(assigns) do
    ~H"""
    <h1>Products</h1>
    <ul>
      <li :for={product <- @products}>
        {product.name} - ${product.price}
      </li>
    </ul>
    <button phx-click="refresh">Refresh</button>
    """
  end

  def handle_event("refresh", _params, socket) do
    # Called when user clicks the button - no page reload!
    products = Hello.Store.list_products()
    {:noreply, assign(socket, products: products)}
  end
end
```

### What Happens Under the Hood

1. **Initial request** - Server renders HTML, sends to browser
2. **WebSocket connects** - `socket "/live"` from endpoint.ex
3. **LiveView process starts** - Keeps state in server memory
4. **User clicks button** - Event sent via WebSocket (not HTTP!)
5. **Server updates state** - Re-renders only changed parts
6. **Browser patches DOM** - Morphdom applies minimal changes

## Side-by-Side Comparison

| Aspect | Controller | LiveView |
|--------|------------|----------|
| **Connection** | Opens, responds, closes | Stays open (WebSocket) |
| **State** | None (stateless) | Kept on server (`assigns`) |
| **Updates** | Full page reload | Partial DOM patches |
| **Good for** | Downloads, APIs, static pages | Forms, dashboards, real-time |
| **Server resources** | Minimal | Process per user |

---

## When to Use Each

### Use Controller for:
- File downloads/uploads
- API endpoints returning JSON
- Static pages that rarely change
- Redirects and authentication flows
- Webhooks from external services

### Use LiveView for:
- Interactive forms with real-time validation
- Search with live filtering
- Dashboards with live updates
- Multi-step wizards
- Shopping carts
- Chat interfaces

## HTTP Verbs Quick Reference

```elixir
scope "/products", HelloWeb do
  pipe_through :browser

  get "/",        ProductController, :index    # List all
  get "/:id",     ProductController, :show     # Show one
  get "/new",     ProductController, :new      # Form to create
  post "/",       ProductController, :create   # Submit create
  get "/:id/edit", ProductController, :edit    # Form to edit
  put "/:id",     ProductController, :update   # Submit edit (full)
  patch "/:id",   ProductController, :update   # Submit edit (partial)
  delete "/:id",  ProductController, :delete   # Delete
end

# Or use resources/4 to generate all of these:
resources "/products", ProductController
```

## Summary

1. **Endpoint** - Processes ALL requests through plugs (middleware)
2. **Router** - Matches URL to either Controller or LiveView
3. **Controller** - One-time request/response, then connection closes
4. **LiveView** - Persistent WebSocket, server keeps state, real-time updates

**Rule of thumb:** Start with LiveView for interactive pages. Use Controller for file operations, APIs, and simple static pages.
