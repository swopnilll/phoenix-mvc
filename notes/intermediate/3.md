# Phoenix Router — A Practical Guide for React Developers

## Part 2: Database Calls and LiveView

In React you'd probably `useEffect` + `fetch` to load data from an API on mount. In LiveView, there's no API layer needed. The LiveView process runs on the server — it can just call the database directly.

Let's make our Products page load a real list of products.

---

### The Layers — How Phoenix Structures Data Access

Before writing code, understand the three layers Phoenix uses:

```
Browser / LiveView  ←→  Context (your "API")  ←→  Repo (database)
```

| Layer | File location | What it does |
|---|---|---|
| **LiveView** | `live/products_live.ex` | UI logic, handles events, holds state |
| **Context** | `lib/ordering_system/catalog.ex` | Business logic, data fetching functions |
| **Repo** | Built-in via Ecto | Runs the actual SQL queries |
| **Schema** | `lib/ordering_system/catalog/product.ex` | Defines the shape of a database table |

> **React equivalent of this layering:**
> - LiveView = your React component
> - Context = your API service layer (`/api/products.js`)
> - Repo = your database ORM (Prisma, Sequelize, etc.)
> - Schema = your TypeScript type / database model

### Step 5 — Create the Product Schema

The schema maps an Elixir struct to a database table. Think of it as your TypeScript type + database model in one.

Create `lib/ordering_system/catalog/product.ex`:

```elixir
defmodule OrderingSystem.Catalog.Product do
  use Ecto.Schema

  schema "products" do           # maps to the "products" table in Postgres
    field :name,        :string
    field :price,       :decimal
    field :description, :string

    timestamps()                 # adds inserted_at and updated_at columns
  end
end
```

> This is equivalent to a Prisma model:
> ```prisma
> model Product {
>   id          Int      @id @default(autoincrement())
>   name        String
>   price       Decimal
>   description String
>   createdAt   DateTime @default(now())
>   updatedAt   DateTime @updatedAt
> }
> ```

### Step 6 — Create the Context (your data access layer)

The Context is the function boundary between your UI and your database. You never call `Repo` directly from a LiveView — you call the Context, and the Context calls `Repo`.

Create `lib/ordering_system/catalog.ex`:

```elixir
defmodule OrderingSystem.Catalog do
  alias OrderingSystem.Repo
  alias OrderingSystem.Catalog.Product

  # Fetch all products from the database
  def list_products do
    Repo.all(Product)
  end

  # Fetch a single product by ID — returns nil if not found
  def get_product(id) do
    Repo.get(Product, id)
  end

  # Delete a product by ID
  def delete_product(id) do
    product = Repo.get(Product, id)
    Repo.delete(product)
  end
end
```

`Repo.all(Product)` generates and runs `SELECT * FROM products`. That's it. No SQL string, no connection management — Ecto handles it.

> **React equivalent:** This is your fetch wrapper:
> ```js
> // catalog.js
> export async function listProducts() {
>   const res = await fetch('/api/products')
>   return res.json()
> }
> ```
> Except here there's no HTTP hop. It's a direct in-process function call to the database.

### Step 7 — Load Data in the LiveView

Now let's wire the Products LiveView to actually fetch and display products.

Update `lib/ordering_system_web/live/products_live.ex`:

```elixir
defmodule OrderingSystemWeb.ProductsLive do
  use OrderingSystemWeb, :live_view

  alias OrderingSystem.Catalog   # bring in our context

  # mount/3 runs when the LiveView first loads — like useEffect([], []) on mount
  def mount(_params, _session, socket) do
    products = Catalog.list_products()   # direct DB call, no fetch/await needed

    socket = assign(socket, :products, products)
    # assign() puts data into the socket — like setState in React
    # :products is the key, products is the value

    {:ok, socket}
  end

  def render(assigns) do
    ~H"""
    <h1>Products</h1>

    <ul>
      <%= for product <- @products do %>
        <li>
          <strong><%= product.name %></strong> — $<%= product.price %>
          <p><%= product.description %></p>
        </li>
      <% end %>
    </ul>
    """
  end
end
```

#### What just happened?

```
1. User visits /products
2. Phoenix starts a LiveView process on the server
3. mount/3 runs — calls Catalog.list_products() — hits the DB
4. Products are stored in the socket via assign()
5. render/1 runs — @products is the list — HTML is generated
6. HTML is sent to the browser over WebSocket
7. The process stays alive, connected, ready to push updates
```

> **React mental model:**
> ```jsx
> function ProductsPage() {
>   const [products, setProducts] = useState([])
>
>   useEffect(() => {
>     fetch('/api/products')
>       .then(r => r.json())
>       .then(data => setProducts(data))
>   }, [])
>
>   return (
>     <ul>
>       {products.map(p => <li key={p.id}>{p.name} — ${p.price}</li>)}
>     </ul>
>   )
> }
> ```
> The difference: no API route, no `fetch`, no `useEffect`. `mount` IS the effect. `assign` IS `setState`. And it all runs on the server.

---

### Step 8 — The "assigns" concept in depth

`assign(socket, :products, products)` is the core of how LiveView holds state. The socket is like a box that holds your component's state — and `assigns` is the map inside that box.

```elixir
# You can assign multiple things at once using the pipe operator:
socket =
  socket
  |> assign(:products, products)
  |> assign(:loading, false)
  |> assign(:page_title, "Products")

# In the template, access them with @
# @products  ==  socket.assigns.products
# @loading   ==  socket.assigns.loading
```

In React terms, every `assign` is like a `setState` call — but instead of triggering a re-render of the whole React tree, Phoenix diffs only the changed HTML and sends the minimal patch down the WebSocket.

### Step 9 — Handling a User Event (Delete a Product)

LiveView isn't just for displaying data. It handles user interactions too — without writing a single API route.

Let's add a delete button to each product.

```elixir
defmodule OrderingSystemWeb.ProductsLive do
  use OrderingSystemWeb, :live_view

  alias OrderingSystem.Catalog

  def mount(_params, _session, socket) do
    products = Catalog.list_products()
    {:ok, assign(socket, :products, products)}
  end

  # handle_event/3 runs when the user triggers a phx-click, phx-submit, etc.
  # "delete_product" matches the phx-click value in the template below
  def handle_event("delete_product", %{"id" => id}, socket) do
    Catalog.delete_product(id)            # delete from DB
    products = Catalog.list_products()    # reload the list
    {:noreply, assign(socket, :products, products)}
    # {:noreply, ...} means "no reply to the browser needed, just update the state"
  end

  def render(assigns) do
    ~H"""
    <h1>Products</h1>

    <ul>
      <%= for product <- @products do %>
        <li>
          <strong><%= product.name %></strong> — $<%= product.price %>
          <button phx-click="delete_product" phx-value-id={product.id}>
            Delete
          </button>
        </li>
      <% end %>
    </ul>
    """
  end
end
```

#### The flow when Delete is clicked:

```
1. User clicks "Delete" button
2. phx-click sends "delete_product" event over the existing WebSocket
3. handle_event/3 runs on the server
4. DB row is deleted, list is reloaded
5. assign() updates the socket state
6. Phoenix diffs the old HTML vs new HTML
7. Only the changed part (the removed <li>) is sent down the socket
8. Browser patches the DOM — no page reload, no fetch call
```

> **React equivalent:**
> ```jsx
> async function handleDelete(id) {
>   await fetch(`/api/products/${id}`, { method: 'DELETE' })
>   const updated = await fetch('/api/products').then(r => r.json())
>   setProducts(updated)
> }
> <button onClick={() => handleDelete(product.id)}>Delete</button>
> ```
> In LiveView: no API route to write, no fetch calls, no async/await. The button fires an event name over the socket and the server handles it.

### The Full Picture — What We've Built

```
router.ex
├── live "/", HomeLive
└── live "/products", ProductsLive
                          │
                          ├── mount()         → calls Catalog.list_products()
                          ├── handle_event()  → calls Catalog.delete_product()
                          └── render()        → displays @products
                                                      │
                                              Catalog (context)
                                                      │
                                              Repo (Ecto / Postgres)
                                                      │
                                              products table
```

The router is just the entry point. The real work happens in the LiveView process — which lives on the server, holds state in `assigns`, reacts to events, and pushes minimal HTML diffs to the browser over a single persistent WebSocket connection.
