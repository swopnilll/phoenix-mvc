## Node.js vs Phoenix: Two Different Approaches

### Node.js/Express (Organized by Type)

```
src/
в”ңв”Җв”Җ controllers/
в”Ӯ   в”ңв”Җв”Җ userController.js
в”Ӯ   в”ңв”Җв”Җ productController.js
в”Ӯ   в””в”Җв”Җ orderController.js
в”ңв”Җв”Җ models/
в”Ӯ   в”ңв”Җв”Җ User.js
в”Ӯ   в”ңв”Җв”Җ Product.js
в”Ӯ   в””в”Җв”Җ Order.js
в”ңв”Җв”Җ services/
в”Ӯ   в”ңв”Җв”Җ userService.js
в”Ӯ   в”ңв”Җв”Җ productService.js
в”Ӯ   в””в”Җв”Җ orderService.js
в”ңв”Җв”Җ middlewares/
в”Ӯ   в””в”Җв”Җ auth.js
в””в”Җв”Җ routes/
    в””в”Җв”Җ index.js
```

**Problem:** To understand "Products", you look in 4+ folders.

### Phoenix (Organized by Domain)

```
lib/
в”ңв”Җв”Җ hello_web/                    # Web stuff (like controllers/)
в”Ӯ   в”ңв”Җв”Җ router.ex
в”Ӯ   в””в”Җв”Җ live/
в”Ӯ       в””в”Җв”Җ products_live.ex
в”Ӯ
в””в”Җв”Җ hello/                        # Business logic
    в”ңв”Җв”Җ catalog.ex                # Products context
    в”ңв”Җв”Җ catalog/
    в”Ӯ   в”ңв”Җв”Җ product.ex            # Product model
    в”Ӯ   в””в”Җв”Җ policy.ex             # Product permissions
    в”Ӯ
    в”ңв”Җв”Җ accounts.ex               # Users context
    в””в”Җв”Җ accounts/
        в””в”Җв”Җ user.ex               # User model
```

**Benefit:** Everything about "Products" is in `catalog.ex` + `catalog/`.

## The Two Folders Explained

### `lib/hello_web/` - The Web Layer

**What goes here:** Anything that deals with HTTP requests/responses.

```
lib/hello_web/
в”ңв”Җв”Җ router.ex              # URL routes (like Express router)
в”ңв”Җв”Җ endpoint.ex            # Server config
в”ңв”Җв”Җ controllers/           # Traditional controllers
в”ңв”Җв”Җ live/                  # LiveView components
в”Ӯ   в”ңв”Җв”Җ home_live.ex
в”Ӯ   в””в”Җв”Җ products_live.ex
в”ңв”Җв”Җ components/            # Reusable UI components
в””в”Җв”Җ layouts/               # HTML layouts
```

**Rule:** This folder should NOT contain business logic. Only:
- Routing
- Rendering HTML
- Handling HTTP requests
- Calling Context functions

### `lib/hello/` - The Business Logic

**What goes here:** All your actual application logic.

```
lib/hello/
в”ңв”Җв”Җ catalog.ex             # Context: "Catalog" domain
в”ңв”Җв”Җ catalog/
в”Ӯ   в”ңв”Җв”Җ product.ex         # Schema: Product table
в”Ӯ   в””в”Җв”Җ policy.ex          # Policy: Who can do what
в”Ӯ
в”ңв”Җв”Җ accounts.ex            # Context: "Accounts" domain
в”ңв”Җв”Җ accounts/
в”Ӯ   в””в”Җв”Җ user.ex            # Schema: User table
в”Ӯ
в””в”Җв”Җ repo.ex                # Database connection
```

**Rule:** This folder should NOT know about HTTP, HTML, or web stuff.

## What is a "Context"?

A Context is a **module that groups related functionality**.

```elixir
# lib/hello/catalog.ex

defmodule Hello.Catalog do
  @moduledoc """
  The Catalog context.
  Everything about products goes through here.
  """

  alias Hello.Repo
  alias Hello.Catalog.Product

  # Public API - what the outside world can call

  def list_products do
    Repo.all(Product)
  end

  def get_product(id) do
    Repo.get(Product, id)
  end

  def create_product(attrs) do
    %Product{}
    |> Product.changeset(attrs)
    |> Repo.insert()
  end

  def delete_product(id) do
    product = Repo.get(Product, id)
    Repo.delete(product)
  end
end
```

### JS Equivalent

```javascript
// services/CatalogService.js

class CatalogService {
  async listProducts() {
    return await Product.findAll();
  }

  async getProduct(id) {
    return await Product.findByPk(id);
  }

  async createProduct(attrs) {
    return await Product.create(attrs);
  }

  async deleteProduct(id) {
    const product = await Product.findByPk(id);
    return await product.destroy();
  }
}

module.exports = new CatalogService();
```

## The File + Folder Convention

This confuses many people:

```
lib/hello/
в”ңв”Җв”Җ catalog.ex           # вҶҗ Module: Hello.Catalog
в””в”Җв”Җ catalog/             # вҶҗ Folder for sub-modules
    в”ңв”Җв”Җ product.ex       # вҶҗ Module: Hello.Catalog.Product
    в””в”Җв”Җ policy.ex        # вҶҗ Module: Hello.Catalog.Policy
```

### The Rule

| File/Folder | Module Name |
|-------------|-------------|
| `catalog.ex` | `Hello.Catalog` |
| `catalog/product.ex` | `Hello.Catalog.Product` |
| `catalog/policy.ex` | `Hello.Catalog.Policy` |

The **folder name** = the **parent module namespace**.

### Why?

It's like classes and nested classes in OOP:

```javascript
// Conceptually similar to:
class Catalog {
  static Product = class { ... }
  static Policy = class { ... }
}
```

## Comparison Table

| Layer | Phoenix | Express/Node.js | Purpose |
|-------|---------|-----------------|---------|
| Router | `router.ex` | `routes/index.js` | URL вҶ’ Handler |
| Handler | LiveView / Controller | Controller | Handle request |
| Service | Context (`catalog.ex`) | Service class | Business logic |
| Model | Schema (`product.ex`) | Sequelize Model | DB structure |
| Auth | Policy (`policy.ex`) | Middleware | Permissions |

---

## Why This Architecture?

### 1. Clear Boundaries

```
вқҢ Bad: LiveView directly queries database
products = Repo.all(Product)  # Don't do this in LiveView!

вң… Good: LiveView calls Context
products = Catalog.list_products()  # Context handles DB
```

### 2. Easy Testing

```elixir
# Test the context in isolation (no HTTP needed)
test "list_products returns all products" do
  Catalog.create_product(%{name: "Test", price: 10})
  assert length(Catalog.list_products()) == 1
end
```

### 3. Reusability

```elixir
# Same context used by:
# - LiveView
# - REST API Controller
# - GraphQL Resolver
# - Background Job
# - CLI Script

products = Catalog.list_products()  # Works everywhere
```

### 4. Domain Organization

When someone asks "where's the product code?":

```
Phoenix: "Look in lib/hello/catalog.ex and lib/hello/catalog/"
Express: "Look in models/, services/, controllers/, routes/..."
```

## Quick Rules

1. **LiveViews/Controllers** вҶ’ Only call Context functions
2. **Contexts** вҶ’ Handle all business logic, call Repo
3. **Schemas** вҶ’ Define database structure only
4. **Policies** вҶ’ Define authorization rules only

## TL;DR

| Question | Answer |
|----------|--------|
| What is `catalog.ex`? | The "service" for all product operations |
| What is `catalog/`? | Folder for sub-modules (schema, policy) |
| Why both? | `.ex` = public API, folder = internal modules |
| Why not like Node.js? | Groups by **domain**, not by **file type** |
| Who calls Context? | LiveViews, Controllers, APIs, Jobs |
| Who calls Schema directly? | Only the Context (usually) |
