# Bodyguard Authorization - Complete Guide for JS Developers

## What is Bodyguard?

Bodyguard is an **authorization library** for Elixir. It answers one question:

> "Can this user perform this action?"

### JS Equivalent

```javascript
// Express middleware approach
const canDeleteProduct = (user) => {
  return user.role === 'admin';
};

// Usage
if (canDeleteProduct(req.user)) {
  // allow delete
} else {
  res.status(403).json({ error: 'Forbidden' });
}
```

```elixir
# Elixir Bodyguard approach
if Bodyguard.permit?(Catalog, :delete_product, current_user) do
  # allow delete
else
  # redirect or show error
end
```

## The Mental Model

Think of Bodyguard as a **security guard** at a building:

```
User wants to do something
        │
        ▼
┌─────────────────────────────────┐
│  Bodyguard.permit?(Module,      │
│                    :action,     │
│                    user)        │
└─────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────┐
│  Policy Module checks rules     │
│  "Is this user allowed?"        │
└─────────────────────────────────┘
        │
        ▼
    true / false
```

## The Three Pieces

### 1. Policy Module (The Rules)

**Location:** `lib/hello/catalog/policy.ex`

This is where you define WHO can do WHAT.

```elixir
defmodule Hello.Catalog.Policy do
  @behaviour Bodyguard.Policy          # Tells Elixir "this is a policy"

  alias Hello.Accounts.User

  # Pattern: def authorize(:action, user, params)

  # Admin can delete products
  def authorize(:delete_product, %User{role: "admin"}, _params), do: true

  # Everyone else cannot
  def authorize(:delete_product, _user, _params), do: false
end
```

**JS equivalent:**
```javascript
const CatalogPolicy = {
  authorize(action, user, params) {
    if (action === 'delete_product') {
      if (user.role === 'admin') return true;
      return false;
    }
    return false;
  }
};
```

---

### 2. Context Module (The Bridge)

**Location:** `lib/hello/catalog.ex`

This connects your context to the policy.

```elixir
defmodule Hello.Catalog do
  # This ONE line connects Bodyguard to your Policy
  defdelegate authorize(action, user, params), to: Hello.Catalog.Policy

  # ... rest of your context functions
end
```

**What `defdelegate` means:**

```elixir
# This:
defdelegate authorize(action, user, params), to: Hello.Catalog.Policy

# Is the same as writing:
def authorize(action, user, params) do
  Hello.Catalog.Policy.authorize(action, user, params)
end
```

**JS equivalent:**
```javascript
class Catalog {
  // Delegate to policy
  static authorize(action, user, params) {
    return CatalogPolicy.authorize(action, user, params);
  }
}
```

## Complete Flow Diagram

```
# When you call:
Bodyguard.permit?(Catalog, :delete_product, current_user)

# Step 1: Bodyguard looks for authorize/3 in Catalog module
Catalog.authorize(:delete_product, current_user, %{})

# Step 2: Catalog delegates to Policy (because of defdelegate)
Hello.Catalog.Policy.authorize(:delete_product, current_user, %{})

# Step 3: Policy pattern-matches to find the right rule
def authorize(:delete_product, %User{role: "admin"}, _params), do: true
#                              ↑
#                     Does user.role match "admin"?
#                              │
#                     Yes → return true
#                     No  → try next function clause

# Step 4: Returns true or false
```

## Pattern Matching in Policies (The Magic)

This is what makes Elixir elegant. Instead of if/else chains:

**JS approach (verbose):**
```javascript
function authorize(action, user, params) {
  if (action === 'list_products') {
    if (user.role === 'admin' || user.role === 'manager') {
      return true;
    }
    return false;
  }
  if (action === 'delete_product') {
    if (user.role === 'admin') {
      return true;
    }
    return false;
  }
  return false;
}
```

**Elixir approach (pattern matching):**
```elixir
# Rule 1: list_products for admin
def authorize(:list_products, %User{role: "admin"}, _), do: true

# Rule 2: list_products for manager
def authorize(:list_products, %User{role: "manager"}, _), do: true

# Rule 3: list_products for anyone else
def authorize(:list_products, _, _), do: false

# Rule 4: delete_product for admin only
def authorize(:delete_product, %User{role: "admin"}, _), do: true

# Rule 5: delete_product for anyone else
def authorize(:delete_product, _, _), do: false

# Catch-all: anything else is denied
def authorize(_, _, _), do: false
```

### How Pattern Matching Works

```elixir
def authorize(:delete_product, %User{role: "admin"}, _params)
#            ↑                 ↑                      ↑
#            │                 │                      └── Ignore params (underscore = don't care)
#            │                 └── Match User struct where role is "admin"
#            └── Match atom :delete_product
```

Elixir tries each function from top to bottom until one matches.

## Syntax Breakdown for JS Developers

### 1. The `%User{role: "admin"}` syntax

This is **pattern matching** on a struct:

```elixir
# Elixir
%User{role: "admin"}

# Means: "A User struct where role equals 'admin'"
# Similar to JS destructuring:
const { role } = user;
if (role === 'admin') { ... }
```

### 2. The underscore `_`

Underscore means "I don't care about this value":

```elixir
def authorize(:list_products, _user, _params), do: false
#                             ↑       ↑
#                    "any user"  "any params"
```

### 3. The `do: true` syntax

Short form of a function body:

```elixir
# Short form
def authorize(:delete_product, %User{role: "admin"}, _), do: true

# Same as long form
def authorize(:delete_product, %User{role: "admin"}, _) do
  true
end
```

### 4. The `@behaviour` line

```elixir
@behaviour Bodyguard.Policy
```

This is like **implementing an interface** in TypeScript:

```typescript
interface Policy {
  authorize(action: string, user: User, params: any): boolean;
}

class CatalogPolicy implements Policy {
  authorize(action, user, params) { ... }
}
```

## Common Patterns

### Multiple roles allowed

```elixir
# Using guard clause with "when"
def authorize(:list_products, %User{role: role}, _)
    when role in ["admin", "manager"],
    do: true
```

### Check ownership

```elixir
# User can only edit their own profile
def authorize(:edit_profile, %User{id: user_id}, %{profile_id: profile_id})
    when user_id == profile_id,
    do: true
```

### Pass resource in params

```elixir
# In LiveView
Bodyguard.permit?(Catalog, :edit_product, user, product)

# In Policy
def authorize(:edit_product, %User{role: "admin"}, _product), do: true
def authorize(:edit_product, %User{id: user_id}, %Product{owner_id: owner_id})
    when user_id == owner_id,
    do: true
```

## Summary

1. **Policy** = The rules (who can do what)
2. **defdelegate** = Connects context to policy
3. **Bodyguard.permit?** = Asks "can this user do this?"
4. **Pattern matching** = Elixir matches the right rule automatically

It's cleaner than if/else chains because each rule is a separate function!
