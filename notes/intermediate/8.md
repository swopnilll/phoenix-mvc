# Understanding @spec in Elixir (Type Specifications)

## The Problem: Elixir is Dynamically Typed

```elixir
# Elixir doesn't care about types at runtime
def add(a, b) do
  a + b
end

add(1, 2)       # Works: 3
add("a", "b")   # Works: "ab"
add(1, "b")     # CRASHES at runtime!
```

```javascript
// Same in JavaScript
function add(a, b) {
  return a + b;
}

add(1, 2)       // 3
add("a", "b")   // "ab"
add(1, "b")     // "1b" (weird coercion)
```

---

## The Solution: @spec (Type Specifications)

`@spec` is like **TypeScript for Elixir** - but optional documentation that tools can check.

```elixir
@spec add(integer(), integer()) :: integer()
def add(a, b) do
  a + b
end
```

This says: *"add takes two integers and returns an integer"*

---

## Syntax Breakdown

```elixir
@spec function_name(arg1_type, arg2_type) :: return_type

@spec add(integer(), integer()) :: integer()
#     │    │          │           │
#     │    │          │           └── Returns an integer
#     │    │          └── Second arg is integer
#     │    └── First arg is integer
#     └── Function name
```

---

## Comparison: TypeScript vs Elixir @spec

### TypeScript

```typescript
function greet(name: string): string {
  return `Hello, ${name}`;
}

// Types are INLINE with the code
```

### Elixir

```elixir
@spec greet(String.t()) :: String.t()
def greet(name) do
  "Hello, #{name}"
end

# Types are ABOVE the function as an attribute
```

## Common Types in Elixir

| Type | Example Values |
|------|----------------|
| `integer()` | `1`, `42`, `-10` |
| `float()` | `3.14`, `-0.5` |
| `number()` | Any integer or float |
| `boolean()` | `true`, `false` |
| `String.t()` | `"hello"` |
| `atom()` | `:ok`, `:error`, `nil` |
| `list(integer())` | `[1, 2, 3]` |
| `map()` | `%{key: "value"}` |
| `tuple()` | `{:ok, "result"}` |
| `any()` | Anything |

## Real Examples

### Example 1: Simple function

```elixir
@spec authorize(atom(), User.t(), map()) :: boolean()
def authorize(:list_pricing, %User{role: role}, _params)
    when role in [:admin, :operations_manager],
    do: true
```

**Translation:**
- Takes: an atom, a User struct, and a map
- Returns: `true` or `false`

---

### Example 2: LiveView mount

```elixir
alias Phoenix.LiveView.Socket

@spec mount(map(), map(), Socket.t()) :: {:ok, Socket.t()}
def mount(_params, _session, socket) do
  {:ok, socket |> assign(:page_title, "Pricing")}
end
```

**Translation:**
- Takes: params (map), session (map), socket (Socket struct)
- Returns: tuple of `:ok` and a Socket

---

### Example 3: Union types (multiple possible returns)

```elixir
@spec divide(number(), number()) :: {:ok, float()} | {:error, String.t()}
def divide(_a, 0), do: {:error, "Cannot divide by zero"}
def divide(a, b), do: {:ok, a / b}
```

**Translation:**
- Returns EITHER `{:ok, float}` OR `{:error, string}`
- The `|` means "or"

## Why Use @spec?

### 1. Documentation

```elixir
# Without @spec - what does this return?
def process(data) do
  # ... 50 lines of code
end

# With @spec - immediately clear!
@spec process(map()) :: {:ok, list()} | {:error, String.t()}
def process(data) do
  # ... 50 lines of code
end
```

### 2. Dialyzer (Static Analysis Tool)

```bash
# Run type checking
mix dialyzer
```

Dialyzer reads your `@spec` and checks if your code actually matches:

```elixir
@spec add(integer(), integer()) :: integer()
def add(a, b) do
  "not a number"  # Dialyzer ERROR: spec says integer, you return string!
end
```

### 3. Editor Support

With ElixirLS in VS Code, you get:
- Hover info showing types
- Autocomplete with type hints
- Warnings when types don't match

## The `.t()` Syntax Explained

```elixir
alias Phoenix.LiveView.Socket

@spec authorize(Socket.t()) :: Socket.t()
```

### What is `.t()`?

By convention, Elixir modules define a `@type t` for their main struct:

```elixir
defmodule Phoenix.LiveView.Socket do
  @type t :: %__MODULE__{
    assigns: map(),
    # ... other fields
  }

  defstruct [:assigns, :endpoint, ...]
end
```

So `Socket.t()` means *"the main type defined by the Socket module"* (the socket struct).

---

## With vs Without Alias

```elixir
# Without alias - verbose
@spec authorize(Phoenix.LiveView.Socket.t()) :: Phoenix.LiveView.Socket.t()

# With alias - cleaner
alias Phoenix.LiveView.Socket
@spec authorize(Socket.t()) :: Socket.t()
```

## Should You Use @spec?

| Situation | Recommendation |
|-----------|----------------|
| Learning Elixir | Optional - focus on learning first |
| Personal project | Optional - nice to have |
| Team project | Recommended - helps others understand |
| Library/Package | Required - users need to know types |

```elixir
defmodule HelloWeb.ProductsLive do
  use HelloWeb, :live_view

  alias Phoenix.LiveView.Socket

  @spec mount(map(), map(), Socket.t()) :: {:ok, Socket.t()}
  def mount(_params, _session, socket) do
    # ...
  end

  @spec authorize(Socket.t()) :: Socket.t()
  defp authorize(socket) do
    # ...
  end

  @spec handle_event(String.t(), map(), Socket.t()) :: {:noreply, Socket.t()}
  def handle_event("delete", _params, socket) do
    # ...
  end
end
```

## Summary

| Concept | JavaScript/TypeScript | Elixir |
|---------|----------------------|--------|
| Type annotations | Inline: `function(x: number): number` | Above: `@spec func(integer()) :: integer()` |
| Type checker | TypeScript compiler | Dialyzer |
| Required? | TS: Yes, JS: No | No (but recommended) |
| Runtime check? | TS: No (compiled away) | No (documentation only) |

## Quick Reference

```elixir
# Basic types
@spec func(integer()) :: integer()
@spec func(String.t()) :: String.t()
@spec func(boolean()) :: boolean()
@spec func(atom()) :: atom()

# Collections
@spec func(list(integer())) :: list(integer())
@spec func(map()) :: map()

# Tuples (common patterns)
@spec func() :: {:ok, any()} | {:error, String.t()}
@spec func() :: :ok | {:error, term()}

# Structs
@spec func(User.t()) :: User.t()
@spec func(Socket.t()) :: {:ok, Socket.t()}

# Multiple args
@spec func(String.t(), integer(), map()) :: boolean()

# Optional/default args - use multiple specs or \\
@spec func(String.t(), integer()) :: String.t()
```

---

## TL;DR

**`@spec` = Optional type documentation that Dialyzer can verify at compile time.**

```elixir
@spec function_name(arg_types) :: return_type
def function_name(args) do
  # implementation
end
```

It doesn't change runtime behavior - it's purely for documentation and static analysis.
