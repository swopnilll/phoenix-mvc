# use, require, import, alias - The Four Module Keywords

## The Problem

In Elixir, you often need to:
- Use code from other modules
- Avoid typing long module names repeatedly
- Access macros from other modules
- Inject functionality into your module

Elixir has **4 different keywords** for this. Here's what each does:

---

## Quick Overview

```elixir
defmodule AscendWeb.OrderingLive do
  use AscendWeb, :live_view                    # Injects code (like inheritance)

  require Logger                               # Enables macros from Logger

  import AscendWeb.Components.Controls         # Brings functions into scope

  alias Ascend.Ordering.{Order, LineItem}      # Shortens module names

  # Now you can use:
  # - LiveView functions (from use)
  # - Logger.info "message" (from require)
  # - button(), input() (from import)
  # - Order, LineItem (from alias)
end
```

---

## 1. `alias` - Shortening Names

**Purpose:** Create a shortcut for a long module name.

### Without alias

```elixir
def create_order do
  Ascend.Ordering.Order.changeset(%Ascend.Ordering.Order{}, attrs)
  |> Ascend.Repo.insert()
end
```

### With alias

```elixir
alias Ascend.Ordering.Order

def create_order do
  Order.changeset(%Order{}, attrs)  # Much cleaner!
  |> Repo.insert()
end
```

### JS Equivalent

```javascript
// JavaScript import with rename
import { VeryLongModuleName as Short } from './module';

// Usage
Short.doSomething();
```

```elixir
# Elixir alias
alias Ascend.Ordering.VeryLongModuleName, as: Short

# Usage
Short.do_something()
```

### Multiple aliases

```elixir
# Long way
alias Ascend.Ordering.Order
alias Ascend.Ordering.LineItem
alias Ascend.Ordering.Payment

# Short way (same result)
alias Ascend.Ordering.{Order, LineItem, Payment}
```

### Custom alias name

```elixir
alias Ascend.Ordering.Order, as: O

O.changeset(%O{}, attrs)  # Works, but less readable
```

## 2. `import` - Bringing Functions Into Scope

**Purpose:** Call functions WITHOUT the module prefix.

### Without import

```elixir
def render(assigns) do
  AscendWeb.Components.Controls.button("Click me")
  AscendWeb.Components.Controls.input(type: "text")
  AscendWeb.Components.Controls.select(options: @options)
end
```

### With import

```elixir
import AscendWeb.Components.Controls

def render(assigns) do
  button("Click me")       # No prefix needed!
  input(type: "text")
  select(options: @options)
end
```

### JS Equivalent

```javascript
// JavaScript named imports
import { button, input, select } from './Controls';

// Usage - no prefix
button("Click me");
```

```elixir
# Elixir import
import AscendWeb.Components.Controls

# Usage - no prefix
button("Click me")
```

### Selective import

```elixir
# Only import specific functions
import AscendWeb.Components.Controls, only: [button: 1, input: 1]
#                                           name: arity (number of args)

# Import everything EXCEPT certain functions
import AscendWeb.Components.Controls, except: [deprecated_func: 2]
```

### Warning: Use Sparingly!

```elixir
# BAD - Where does `process` come from??
import ModuleA
import ModuleB
import ModuleC

def do_work do
  process(data)  # Which module?? Confusing!
end

# GOOD - Clear origin
alias Ascend.Ordering.Processor

def do_work do
  Processor.process(data)  # Clear!
end
```

**Rule:** Only `import` when you'll use functions MANY times (like UI components).

## 3. `require` - Enabling Macros

**Purpose:** Make macros from another module available.

### What's a Macro?

Macros are code that generates code at compile time. You can't use them without `require`.

### Example: Logger

```elixir
defmodule Ascend.Ordering do
  require Logger  # MUST require to use Logger macros

  def create_order(attrs) do
    Logger.info("Creating order: #{inspect(attrs)}")
    #      ↑ This is a macro, not a regular function!

    # Without require, you get:
    # ** (CompileError) you must require Logger before invoking the macro Logger.info/1
  end
end
```

### Why Macros Need require

```elixir
# Logger.info is a MACRO that transforms into:
Logger.info("Creating order")

# Becomes at compile time:
if Application.get_env(:logger, :level) <= :info do
  Logger.bare_log(:info, "Creating order", ...)
end
```

The macro does compile-time checks. Regular functions can't do this.

### JS Equivalent

There's no direct equivalent in JS. The closest is:

```javascript
// Babel macros (build-time code generation)
import { macro } from 'some-macro-package';

// The macro runs at BUILD time, not runtime
macro`some template`;
```

### Common modules that need require

```elixir
require Logger              # Logging macros
require Ecto.Query          # Query macros (from, where, select)
require Phoenix.LiveView    # LiveView macros (rarely needed directly)
```

## 4. `use` - Injecting Code (The Power Tool)

**Purpose:** Inject a bunch of code/functionality into your module.

### Example

```elixir
defmodule AscendWeb.OrderingLive do
  use AscendWeb, :live_view   # Magic happens here!

  # After this line, you automatically have:
  # - mount/3 callback available
  # - handle_event/3 callback available
  # - assign/2 helper
  # - ~H sigil for HTML
  # - All Phoenix.LiveView functions
end
```

### What `use` Actually Does

```elixir
# When you write:
use AscendWeb, :live_view

# Elixir calls:
AscendWeb.__using__(:live_view)

# Which typically does:
defmacro __using__(:live_view) do
  quote do
    use Phoenix.LiveView
    import AscendWeb.Components
    alias AscendWeb.Router.Helpers, as: Routes
    # ... lots more setup
  end
end
```

### JS Equivalent

Think of it like **class inheritance + mixins**:

```javascript
// JavaScript
class OrderingLive extends LiveView {
  // Inherits all LiveView methods
}

// With mixins
class OrderingLive extends mix(LiveView, Components, Helpers) {
  // Has everything from all mixed-in classes
}
```

```elixir
# Elixir
defmodule OrderingLive do
  use AscendWeb, :live_view  # Gets LiveView + Components + Helpers
end
```

### Common uses

```elixir
use Phoenix.LiveView           # LiveView functionality
use Ecto.Schema                # Database schema functionality
use GenServer                  # OTP server functionality
use AscendWeb, :controller     # Controller functionality
use AscendWeb, :live_view      # LiveView functionality
```

## Ordering App Example

Let's build an ordering system to see all four in action:

```elixir
defmodule AscendWeb.OrderingLive do
  # ═══════════════════════════════════════════════════════════════
  # USE - Inject LiveView functionality
  # ═══════════════════════════════════════════════════════════════
  use AscendWeb, :live_view
  # Now we have: mount/3, handle_event/3, assign/2, ~H sigil, etc.

  # ═══════════════════════════════════════════════════════════════
  # REQUIRE - Enable Logger macros
  # ═══════════════════════════════════════════════════════════════
  require Logger
  # Now we can use: Logger.info(), Logger.error(), etc.

  # ═══════════════════════════════════════════════════════════════
  # IMPORT - Bring component functions into scope (no prefix needed)
  # ═══════════════════════════════════════════════════════════════
  import AscendWeb.Components.Controls
  # Now we can call: button(), input(), modal() without prefix

  # ═══════════════════════════════════════════════════════════════
  # ALIAS - Shorten long module names
  # ═══════════════════════════════════════════════════════════════
  alias Ascend.Ordering
  alias Ascend.Ordering.{Order, LineItem, Payment}
  alias Phoenix.LiveView.Socket
  # Now we use: Order instead of Ascend.Ordering.Order

  # ═══════════════════════════════════════════════════════════════
  # THE CODE
  # ═══════════════════════════════════════════════════════════════

  @spec mount(map(), map(), Socket.t()) :: {:ok, Socket.t()}
  def mount(_params, _session, socket) do
    Logger.info("OrderingLive mounted")  # require Logger

    orders = Ordering.list_orders()       # alias Ascend.Ordering

    {:ok, assign(socket, orders: orders)} # use AscendWeb, :live_view
  end

  def handle_event("add_item", %{"product_id" => id}, socket) do
    Logger.debug("Adding item: #{id}")    # require Logger

    line_item = %LineItem{product_id: id} # alias Ascend.Ordering.LineItem

    {:noreply, socket}                    # use AscendWeb, :live_view
  end

  def render(assigns) do
    ~H"""
    <div>
      <h1>Orders</h1>

      <%= for order <- @orders do %>
        <div>
          Order #<%= order.id %>
          <%= button("View", phx_click: "view_order") %>
        </div>
      <% end %>

      <%= modal id="order-modal" do %>
        Order details here
      <% end %>
    </div>
    """
    # ~H sigil from: use AscendWeb, :live_view
    # button(), modal() from: import AscendWeb.Components.Controls
  end
end
```

## Comparison Table

| Keyword | Purpose | JS Equivalent | When to Use |
|---------|---------|---------------|-------------|
| `alias` | Shorten name | `import X as Y` | Always - for cleaner code |
| `import` | No prefix needed | `import { fn }` | UI components, frequently used functions |
| `require` | Enable macros | Babel macros | Logger, Ecto.Query |
| `use` | Inject code | `extends` + mixins | Framework setup (LiveView, GenServer) |

## Common Mistakes

### 1. Forgetting require for macros

```elixir
# ERROR
Logger.info("hello")  # ** (CompileError) you must require Logger

# FIX
require Logger
Logger.info("hello")
```

### 2. import pollution

```elixir
# BAD - too many imports, unclear origins
import Ecto.Query
import Ecto.Changeset
import Phoenix.HTML
import Phoenix.HTML.Form

where(query, ...)  # Where does this come from??

# GOOD - explicit
import Ecto.Query, only: [from: 2, where: 2]
```

### 3. Confusing alias with import

```elixir
# alias - still need prefix
alias Ascend.Ordering.Order
Order.create(attrs)  # ✅ Need Order. prefix

# import - no prefix
import Ascend.Ordering.Order
create(attrs)  # ✅ No prefix (but confusing!)
```

## TL;DR

| Keyword | One-liner |
|---------|-----------|
| `alias` | "Let me type `Order` instead of `Ascend.Ordering.Order`" |
| `import` | "Let me call `button()` instead of `Controls.button()`" |
| `require` | "Let me use macros from this module" |
| `use` | "Set up my module with all the Phoenix/LiveView magic" |

```elixir
defmodule MyApp.SomeLive do
  use MyAppWeb, :live_view     # Setup (always first)

  require Logger               # For logging macros

  import MyAppWeb.Components   # For UI components

  alias MyApp.Context          # For shorter names
  alias MyApp.Context.Schema
end
```
