### Understanding mount and assign — The Full Picture

#### What is the socket?

The socket is LiveView's state container. Think of it as combining React's `useState`, `useContext`, and component props into a single object that lives on the server.

```elixir
# The socket is a struct that looks roughly like this under the hood:
%Phoenix.LiveView.Socket{
  assigns: %{
    products: [],
    loading: false,
    page_title: "Import Products",
    # ...anything you assign() ends up here
  },
  # ...internal LiveView connection info (you never touch this part)
}
```

```js
// React mental model — the socket is like:
const socket = {
  assigns: {
    // This is your "state" — accessible in templates as @variable_name
    products: [],
    loading: false,
    page_title: "Import Products"
  }
}
```

#### React Component Lifecycle vs LiveView Lifecycle

```
React                               LiveView
─────                               ────────
function Component() {              def mount(_params, _session, socket) do
  // component initialises            // LiveView initialises

  const [rows, setRows]               socket = assign(socket, :rows, [])
    = useState([])

  const [headers, setHeaders]         socket = assign(socket, :headers, [])
    = useState([])

  return <JSX />                      {:ok, socket}
}                                   end

                                    def render(assigns) do
                                      ~H"""<HTML />"""
                                    end
```

`mount` is the equivalent of your component function body + `useEffect(() => {}, [])` combined. It runs once when the LiveView first loads, on the server.

#### assign — like setState, but immutable

In React, `setState` schedules a mutation on the component's existing state. In Elixir, data is **immutable** — `assign` always returns a brand new socket, leaving the original untouched.

```elixir
# Elixir — assign returns a NEW socket, the original is unchanged
original_socket = socket
new_socket = assign(socket, :rows, [1, 2, 3])

# original_socket.assigns.rows  =>  []
# new_socket.assigns.rows        =>  [1, 2, 3]
# They are two completely different values in memory
```

```js
// React — setState mutates the same state slot
const [rows, setRows] = useState([])
setRows([1, 2, 3])  // same slot, updated value
```

This immutability is a feature, not a quirk. Bugs where state gets accidentally mutated elsewhere simply cannot happen.

#### The pipe operator |>

You'll see `|>` everywhere in Elixir. It takes the result of the left side and passes it as the first argument to the right side — like method chaining without the methods.

```elixir
# These three are IDENTICAL:

# Option 1 — nested (hard to read)
assign(assign(assign(socket, :page_title, "Import"), :rows, []), :headers, [])

# Option 2 — reassigning each time
socket = assign(socket, :page_title, "Import")
socket = assign(socket, :rows, [])
socket = assign(socket, :headers, [])

# Option 3 — pipe operator (what you'll actually write)
socket =
  socket
  |> assign(:page_title, "Import")
  |> assign(:rows, [])
  |> assign(:headers, [])
```

```js
// JavaScript equivalent — similar to method chaining:
const socket = initialSocket
  .assign('page_title', 'Import')
  .assign('rows', [])
  .assign('headers', [])
```

#### Where do assign and mount come from?

When you write `use OrderingSystemWeb, :live_view` at the top of a LiveView, it imports everything from `Phoenix.LiveView` automatically — including `assign`, `allow_upload`, `push_event`, and more.

```elixir
use OrderingSystemWeb, :live_view
# roughly equivalent to:
# import Phoenix.LiveView   -> gives you: assign, allow_upload, push_event, etc.
# import Phoenix.Component  -> gives you: ~H sigil, attr, slot, etc.
```

```js
// React equivalent:
import { useState, useEffect, useRef } from 'react'
// LiveView does this import automatically via `use`
```

#### The return value {:ok, socket}

```elixir
def mount(_params, _session, socket) do
  # ... setup ...
  {:ok, socket}   # <-- what is this?
end
```

This is a **tuple** — Elixir's way of returning multiple values and signalling success or failure. `{:ok, socket}` means "everything went fine, here is the initialised socket." If something went wrong you'd return `{:error, reason}`.

```js
// Similar to the Node callback pattern:
function mount(params, session, callback) {
  if (success) {
    callback(null, socket)       // null = no error
  } else {
    callback(new Error('mount failed'))
  }
}
```

### What is allow_upload?

`allow_upload` is a built-in LiveView function that configures a file upload slot. In React you'd wire up an `<input type="file">`, manually validate file types, check sizes, and manage selected files in state. LiveView does all of that for you in one call.

```elixir
allow_upload(:spreadsheet,         # a name/key for this upload slot
  accept: ~w(.xlsx .xls),          # allowed file extensions
  max_entries: 1,                   # max number of files at once
  max_file_size: 10_000_000         # max size in bytes (10 MB)
)
```

After calling `allow_upload`, the socket's assigns automatically gains an `uploads` key:

```elixir
# socket.assigns now contains:
%{
  uploads: %{
    spreadsheet: %Phoenix.LiveView.UploadConfig{
      name: :spreadsheet,
      accept: [".xlsx", ".xls"],
      max_entries: 1,
      max_file_size: 10_000_000,
      entries: []     # files the user has selected will appear here
    }
  }
}
```

In the template you just use `<.live_file_input upload={@uploads.spreadsheet} />` and Phoenix handles validation, error messages, and progress tracking automatically.

```jsx
// React — you'd do all of this yourself:
function ImportPage() {
  const [selectedFiles, setSelectedFiles] = useState([])

  const handleFileSelect = (e) => {
    const files = Array.from(e.target.files)
    // validate type manually
    // validate size manually
    // validate count manually
    setSelectedFiles(files)
  }

  return <input type="file" accept=".xlsx,.xls" onChange={handleFileSelect} />
}
```

```elixir
# LiveView — one line, all validation handled:
allow_upload(:spreadsheet, accept: ~w(.xlsx .xls), max_entries: 1, max_file_size: 10_000_000)

# In template:
# <.live_file_input upload={@uploads.spreadsheet} />
```

#### The ~w() sigil

You'll notice `~w(.xlsx .xls)` above. This is a shorthand for writing a list of strings without all the quote characters:

```elixir
~w(.xlsx .xls)          # =>  [".xlsx", ".xls"]
~w(apple banana cherry) # =>  ["apple", "banana", "cherry"]
~w(one two three)       # =>  ["one", "two", "three"]
```

### Step 10 — The Product Import LiveView

Now let's put it all together. This LiveView lets a user upload an Excel file and see its contents rendered as a table — no API route, no multer middleware, no S3 presigned URL.

Create `lib/ordering_system_web/live/product_import_live.ex`:

```elixir
defmodule OrderingSystemWeb.ProductImportLive do
  use OrderingSystemWeb, :live_view

  # mount/3 — initialise all state and configure the upload slot
  def mount(_params, _session, socket) do
    socket =
      socket
      |> assign(:page_title, "Import Products")   # sets the browser tab title
      |> assign(:rows, [])                         # table rows from the Excel file
      |> assign(:headers, [])                      # column headers from row 1
      |> assign(:error, nil)                       # any error message to show
      |> allow_upload(:spreadsheet,                # configure the upload slot
           accept: ~w(.xlsx .xls),
           max_entries: 1,
           max_file_size: 10_000_000
         )

    {:ok, socket}
  end

  # handle_event/3 — fires when the user clicks "Import"
  # The file bytes have already been streamed to a temp path by LiveView at this point
  def handle_event("import", _params, socket) do
    # consume_uploaded_entries reads the temp file and lets us process it
    # It runs our anonymous function once per uploaded file
    results =
      consume_uploaded_entries(socket, :spreadsheet, fn %{path: path}, _entry ->
        # path is the temp file path on the server — parse it directly, no fetch needed
        {:ok, table_id} = Xlsxir.multi_extract(path, 0)   # open the first sheet
        rows = Xlsxir.get_list(table_id)                    # get all rows as a list of lists
        Xlsxir.close(table_id)
        {:ok, rows}
      end)

    case results do
      [rows] when length(rows) > 1 ->
        [headers | data_rows] = rows    # first row is headers, rest is data

        {:noreply,
         socket
         |> assign(:headers, headers)
         |> assign(:rows, data_rows)
         |> assign(:error, nil)}

      _ ->
        {:noreply, assign(socket, :error, "Could not parse the file. Is it a valid Excel file?")}
    end
  end

  def render(assigns) do
    ~H"""
    <h1>Import Products</h1>

    <%!-- phx-submit fires "import" when the form is submitted --%>
    <form phx-submit="import">
      <.live_file_input upload={@uploads.spreadsheet} />
      <button type="submit">Import</button>
    </form>

    <%!-- Show any error --%>
    <%= if @error do %>
      <p style="color: red;"><%= @error %></p>
    <% end %>

    <%!-- Show the parsed table once we have data --%>
    <%= if length(@headers) > 0 do %>
      <table>
        <thead>
          <tr>
            <%= for header <- @headers do %>
              <th><%= header %></th>
            <% end %>
          </tr>
        </thead>
        <tbody>
          <%= for row <- @rows do %>
            <tr>
              <%= for cell <- row do %>
                <td><%= cell %></td>
              <% end %>
            </tr>
          <% end %>
        </tbody>
      </table>
    <% end %>
    """
  end
end
```

### The Full Flow — What Happens When a User Uploads a File

```
1.  User visits /products/import
2.  mount/3 runs on the server
      — assigns :rows, :headers, :error as empty
      — allow_upload configures the :spreadsheet slot
      — {:ok, socket} returned, LiveView process is alive
3.  HTML is sent to the browser over WebSocket

4.  User selects an .xlsx file in the file input
      — LiveView's built-in JS streams the file bytes up the WebSocket
      — No page reload, no separate HTTP upload request
      — Phoenix saves the bytes to a temp path on the server

5.  User clicks "Import"
      — phx-submit fires the "import" event over the WebSocket

6.  handle_event("import", ...) runs on the server
      — consume_uploaded_entries reads the temp file at its path
      — Xlsxir parses the Excel bytes into a list of rows
      — assign(:headers) and assign(:rows) update the socket

7.  Phoenix diffs old HTML vs new HTML
      — The <table> is new, so its HTML patch is sent down the socket

8.  Browser patches the DOM
      — Table appears with no page reload, no fetch, no API
```

### Summary Table — All Concepts Covered

| Concept | React | LiveView |
|---|---|---|
| Initialize state | `useState([])` | `assign(socket, :key, [])` |
| Update state | `setRows(newRows)` | `assign(socket, :rows, newRows)` |
| Access state in template | `{rows}` | `@rows` |
| State container | Multiple `useState` hooks | Single `socket.assigns` map |
| Component mount | `useEffect(() => {}, [])` | `def mount(...)` |
| Event handler | `onClick={handleClick}` | `phx-click="event_name"` + `handle_event` |
| Page title | `document.title = "..."` | `assign(socket, :page_title, "...")` |
| File upload config | Manual with `useState` + validation | `allow_upload(...)` |
| Chaining state updates | `.then()` / method chaining | Pipe operator `\|>` |
| Success/failure signal | `try/catch` / callback `(null, data)` | `{:ok, socket}` / `{:error, reason}` |
| File upload endpoint | Must write `/api/parse-excel` route | `consume_uploaded_entries` reads temp path directly |
| String list shorthand | `['.xlsx', '.xls']` | `~w(.xlsx .xls)` |
| Auto-imported helpers | `import { useState } from 'react'` | `use OrderingSystemWeb, :live_view` (automatic) |
